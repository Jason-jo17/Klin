# BYOK (Bring Your Own Key)

Kiln never asks for or stores your API key on any server we control.

- **In the Bench:** keys live only in your browser's IndexedDB, encrypted
  at rest with a non-extractable AES-GCM key generated by the Web Crypto
  API (`apps/bench/lib/byok/keyStore.ts`). Because the encryption key is
  non-extractable, copying the IndexedDB files to another machine yields an
  unusable key handle — but this does not protect against same-origin XSS,
  since the page's own script can always ask the browser to decrypt with
  that key. See `docs/SECURITY.md` §1 for the full threat model.
- **Running a template standalone:** keys live only in that template's
  local `.env.local` (see each template's `.env.local.example`).

## How a key reaches the provider

1. You paste a key into `ByokKeyForm` (`apps/bench/components/keys/`).
   The form fires a cheap "test connection" call through
   `/api/proxy/[provider]` before saving anything.
2. On success, the key is AES-GCM encrypted and written to IndexedDB.
3. When you use a template in the live playground, the Bench reads the key
   back out of IndexedDB and sends it — over HTTPS, in the request body —
   to `/api/templates/[templateId]`, which forwards it straight to the
   provider SDK call and never persists it.
4. `/api/proxy/[provider]` (used for the connection test) follows the same
   rule: stateless pass-through, no persistence, no logging of the key.

## Supported providers

| Provider | Env var (standalone templates) |
|---|---|
| Anthropic | `ANTHROPIC_API_KEY` |
| OpenAI | `OPENAI_API_KEY` |
| Google | `GOOGLE_GENERATIVE_AI_API_KEY` |

## Rate limiting

`/api/proxy/[provider]`, `/api/templates/[templateId]`, and `/api/jury` are
all rate-limited by IP out of the box (`apps/bench/lib/rateLimit.ts`) — 30
requests/minute for the two generation routes, 5 requests/5 minutes for
`/api/jury` (each run launches a headless browser and fires up to 6 model
calls, so it's far more expensive per request). No configuration or
external dependency required; it degrades gracefully by design.

**Known limitation:** the limiter's state is an in-memory `Map`, scoped to
one server process. If you deploy the hosted demo behind more than one
instance (serverless, multi-region), each instance enforces its own limit
independently rather than sharing one global counter. `checkRateLimit`'s
signature (`key, limit, windowMs`) is intentionally store-agnostic — swap
the `Map` for Vercel KV or Upstash behind the same function if you need a
shared counter across instances.
